# NLM Benchmarks

Three dimensions are measured: retrieval quality (does the right session surface when an agent queries?), classifier quality (do the extracted labels, entities, and decisions accurately represent session content?), and consumption quality (does the agent actually use what it retrieves?). Each dimension produces a number with a reproducible method. When a pre-registered ship gate is missed, the number and the decision appear here alongside the positive results.

## LongMemEval-S retrieval

Both runs below evaluate NLM against the public
[LongMemEval-S dataset](https://github.com/xiaowu0162/longmemeval) (500
questions, ICLR 2025) in **body-only** mode: the classifier is bypassed and
raw session transcripts are fed directly to the indexer. Body-only isolates
the retrieval algorithm from classifier choice, making the result comparable
with other published R@5 figures.

Two honest caveats apply to both runs. First, body-only is the
classifier-independent algorithm floor, not the full install path (see
[`docs/methodology-recall-baseline.md`](methodology-recall-baseline.md)).
Second, LongMemEval-S retrieval operates over a small per-instance haystack
of sessions, which is an easier target than a real corpus of thousands of
accumulated sessions.

### Personal 14-month corpus: a different measurement

Separate from LongMemEval-S, retrieval is also measured on the operator's
real 14-month corpus (2,968 sessions) against a curated gold set of 40
decision queries, keyword mode, shipped configuration.
Source: [`reports/recall-experiments/2026-06-11-ranking-phase2.md`](../reports/recall-experiments/2026-06-11-ranking-phase2.md).

| Config | R@5 | R@1 |
|---|---|---|
| Baseline (keyword-only, pre-tiebreaker) | 72.5% (29/40) | 52.5% (21/40) |
| Shipped (metadata tiebreaker) | **90.0% (36/40)** | **70.0% (28/40)** |

Two caveats attach here, not to the LongMemEval-S numbers above. First, the
frontier-classifier caveat: the corpus labels and entities that keyword mode
scores against were generated by a frontier-tier cloud classifier
(`deepseek-v4-flash`, per [`docs/methodology-recall-baseline.md`](methodology-recall-baseline.md));
a smaller local classifier produces different labels and would produce a
different number. Second, this measurement is not independently reproducible
(the corpus is private session history) and the n=40 gold set is small, so
treat the point estimates as indicative rather than precise.

### How to reproduce

```sh
# Step 1: fetch the dataset (idempotent, skips if already cached)
bash scripts/longmemeval/fetch-dataset.sh

# Step 2: build and run (reproduces the headline n=500 body-only run)
npm run build
node dist/scripts/longmemeval/run-harness.js \
  --variant longmemeval_s_cleaned.json \
  --modes keyword,semantic,hybrid \
  --limit 500 \
  --report-dir reports/longmemeval

# npm alias for the same body-only run
npm run bench:longmemeval

# With classifier (attributes classifier contribution to keyword recall)
npm run bench:classifier -- ollama:qwen3.5:4b
npm run bench:classifier -- deepseek:deepseek-v4-flash
```

First run takes 30+ minutes (embeds haystack sessions with `nomic-embed-text`
via Ollama). Subsequent runs are seconds: embeddings cache on disk keyed by
content hash. Full harness options, caching behavior, and comparison tooling
are in [`docs/methodology-recall-baseline.md`](methodology-recall-baseline.md).

## Classifier tiers

The classifier extracts labels, entities, decisions, and open questions from
each session transcript. These fields are what keyword retrieval scores
against: a weaker classifier produces sparser fields, which directly limits
how high keyword R@5 can climb.

Three operating points measured with `nlm eval --classifier` (n=20 shipped
gold fixtures, run 2026-07-02).
Source: [`docs/classifier-tiers.md`](classifier-tiers.md).

| Tier | Model | Schema valid | Label acc | Entity F1 | Decision F1 | Calibration | p50 latency |
|---|---|---|---|---|---|---|---|
| Floor | qwen3.5-4b (local) | 100% | 80% | 48% | 44% | 85% | 24.0s |
| Mid | qwen3.6-35b-a3b (local MoE) | 100% | 80% | 54% | 57% | 95% | 42.2s |
| Mid | gemma-4-26b-a4b-qat (local MoE) | 100% | 75% | 69% | 59% | 80% | 38.4s |
| Cloud | deepseek-v4-flash | 100% | 95% | 66% | 55% | 95% | 9.0s |

```sh
nlm eval --classifier         # score your configured lane against 20 gold fixtures
nlm eval --classifier --json  # full JSON including per-transcript rows
```

For the upgrade path and how to switch classifier lanes, see
[`docs/classifier-tiers.md`](classifier-tiers.md).

## Does the memory get used?

All numbers from
[`reports/usefulness/2026-07-03-consumption-measurements.md`](../reports/usefulness/2026-07-03-consumption-measurements.md).
The usefulness judge is `qwen3.5:4b` locked at the version in
`scripts/eval/lib/usefulness-judge.ts`.

**Ambient injection (per-prompt hook).** 18.2% of automatically injected
context turns out useful in the response that immediately follows. The
remaining 81.8% is off-topic: the recalled sessions do not match the
agent's actual information need on that turn. Most turns carry no
information gap the ambient context could fill.

**Thin-prompt band (n=19).** Restricting scoring to short prompts (those
more likely to carry a genuine information need) did not improve usefulness.
Thin-prompt band rate: 7.9%.

**Context-augmentation A/B (n=40 paired thin-prompt fires).** A paired
comparison of bare recall output vs. richer augmented context showed:

| Condition | Usefulness | Off-topic |
|---|---|---|
| Bare | 28.7% | 67.5% |
| Augmented | 31.3% | 57.5% |
| Delta | +2.5 pts | |

Pre-registered ship gate: +10 pts delta. Observed: +2.5 pts. **Decision:
NOT SHIPPED.**

The pre-registration was locked before any results were observed (see
`docs/superpowers/plans/2026-07-03-wave-u1-consumption-plan.md`). Richer
context marginally improves usefulness but does not clear the threshold.
Publishing the miss is how the method stays honest: a gate that only
appears when it is cleared is not a gate.

**Pull path (explicit agent tool calls).** When the agent pulls memory
rather than receiving ambient injection, usefulness jumps to 72.4%
(n=78 scored genuine pulls from 195 genuine pulls identified from 3,920 raw
MCP source events). Split by tool:

| Tool | usefulness@pull |
|---|---|
| Fact pulls | 88.1% |
| Session pulls | 66.7% |

The 72.4% vs. 18.2% gap is the primary evidence for the pull-first posture.
Fresh installs now default to pull-on-demand: the per-prompt ambient hook is
off by default, and agents retrieve context by issuing explicit
`recall_sessions`, `recall_facts`, `recall_code`, or `recall_workstream`
calls.

## Comparisons

Published memory system results use different LongMemEval variants and
different metrics. This section records what was verified and why no
cross-system ranking appears here.

**Metric incompatibility.** NLM reports **retrieval R@5**: given a query,
does the right session appear in the top 5 retrieved results? End-to-end
**QA accuracy** measures something different: does the final generated
answer to the question match a reference answer, incorporating both
retrieval and generation quality? A system that scores 90% retrieval R@5
and a system that scores 90% QA accuracy on LongMemEval have not
demonstrated the same capability. Presenting them in a shared ranking
column would be dishonest.

**Zep.** Source: "Zep: A Temporal Knowledge Graph Architecture for Agent
Memory," arXiv:2501.13956,
[arxiv.org/abs/2501.13956](https://arxiv.org/abs/2501.13956), accessed
2026-07-03. Benchmark: LongMemEval-S. Metric: **QA answer accuracy**.
Result: 71.2% (gpt-4o). This is end-to-end QA accuracy, not retrieval R@5.
Not ranked against NLM.

**mem0.** Source: mem0 documentation,
[docs.mem0.ai/core-concepts/memory-evaluation](https://docs.mem0.ai/core-concepts/memory-evaluation),
accessed 2026-07-03. Benchmark: LongMemEval (500 questions). Metric: **QA
answer accuracy** (top-200 retrieval budget, mean 6,787 tokens per query).
Result: 94.4% overall. This is end-to-end QA accuracy, not retrieval R@5.
Not ranked against NLM.

**Letta.** No verifiable published LongMemEval result was found from
Letta's own engineering resources. Omitted.

Because no vendor was found publishing LongMemEval-S retrieval R@5, there
is no valid cross-system retrieval ranking to present. To measure a
comparable retrieval number for any memory system, the LongMemEval-S
harness in this repo can be adapted: swap the retrieval function and run
the same scoring code.

## Private-corpus benchmark

A 50-question benchmark on the operator's real 14-month corpus is being
locked, broadening the n=40 decision-query gold set described above.
Categories span six generic classes: decision-recall, status-check,
bug-resolution, config-lookup, temporal, and multi-session. Results will
publish as aggregate scores only; no query content from the private corpus
will appear in the public repo. This benchmark is the only way to attribute
the contribution of classifier tier to end-to-end recall on a production
corpus, and it complements the public LongMemEval-S number rather than
replacing it.

## Reproduce everything

```sh
# Fetch the LongMemEval-S dataset (one-time, idempotent)
bash scripts/longmemeval/fetch-dataset.sh

# Build
npm run build

# Body-only retrieval benchmark (reproduces the headline n=500 run)
node dist/scripts/longmemeval/run-harness.js \
  --variant longmemeval_s_cleaned.json \
  --modes keyword,semantic,hybrid \
  --limit 500 \
  --report-dir reports/longmemeval

# Same run via npm alias
npm run bench:longmemeval

# Rank-detail re-run (n=200, adds R@1/R@3 reporting)
node dist/scripts/longmemeval/run-harness.js \
  --variant longmemeval_s_cleaned.json \
  --modes keyword,hybrid \
  --limit 200 \
  --report-dir reports/longmemeval

# Classifier-in-the-loop (floor tier, local)
npm run bench:classifier -- ollama:qwen3.5:4b

# Classifier-in-the-loop (cloud ceiling, requires DEEPSEEK_API_KEY)
npm run bench:classifier -- deepseek:deepseek-v4-flash

# Compare multiple runs side by side
npm run bench:compare -- \
  reports/longmemeval/<run-1>/results.json \
  reports/longmemeval/<run-2>/results.json

# Classifier quality check against gold fixtures
nlm eval --classifier
```
